# ADR 003: Поиск по товарам — на стороне Python (не в SQL), rapidfuzz-Левенштейн
_Date: 2026-06-02 session 026 | Status: active_

## Context
Нужен поиск по всем товарам с двумя требованиями: (1) регистронезависимость для кириллицы,
(2) терпимость к неполным совпадениям и опечаткам/суффиксам («матрос»→«матрас», «матрасы»→«матрас»).
Текущая БД — SQLite (dev); прод-Postgres только планируется (OPEN).

## Ключевой факт
SQLite `LIKE`/Django `icontains` и SQLite `lower()`/Django `Lower()` регистронезависимы
ТОЛЬКО для латиницы (ASCII). Для кириллицы «Матрас»≠«матрас» и в фильтре, и при попытке
свернуть регистр в SQL. Привести регистр «под капотом» в SQL — нельзя (это и есть корень).
`str.lower()` в Python — полный Unicode. Левенштейна в SQLite тоже нет.

## Options considered
| Option | Pros | Cons |
|--------|------|------|
| ORM `icontains` (Q-OR по полям) | просто, в БД | НЕ регистронезависимо для кириллицы на SQLite; нет опечаток |
| Нормализованная колонка `search_text` (lower в Python) + `contains` | работает на SQLite, в БД | денормализация, синхронизация при правке категории; всё равно нет опечаток |
| **Поиск в Python (выбрано)** | Unicode-регистр, опечатки (Левенштейн), гибкое ранжирование, ноль изменений схемы | грузит активные товары в память; не для огромных каталогов |
| Postgres FTS / pg_trgm | релевантность, стемминг, масштаб | требует миграции на Postgres; сейчас её нет |

## Decision
Поиск реализован в `store/search.py` целиком на Python: нормализация `.lower()`, токенизация,
совпадение = подстрока ИЛИ Левенштейн (через **rapidfuzz**, `score_cutoff`) к словам поля.
Порог «умеренный»: ≤1 правка для слов 4–6 букв, ≤2 для 7+; для слов <4 фуззи выключено.
Ранжирование баллами: точная подстрока > фуззи; вес поля название>категория>описание.
Результат — товары со ВСЕМИ словами (AND); если пусто, отдаём «похожие» (хотя бы одно слово).

## Consequences
- ✅ Кириллица регистронезависима; опечатки/близкие суффиксы ловятся; релевантность настраиваемая.
- ✅ Нулевые изменения моделей/миграций; работает на текущем SQLite.
- ✅ Новая зависимость `rapidfuzz` (быстрый C-Левенштейн). ⚠️ requirements.txt в проекте нет — не забыть при деплое.
- ⚠️ O(товаров×слов) в памяти на каждый запрос — ок для сотен товаров, не для десятков тысяч.
- ⚠️ Морфология покрыта лишь частично (подстрока+фуззи), без настоящего стемминга.

## Confidence
HIGH для текущего масштаба и dev-SQLite. Прямо снимает проблему кириллицы, которую SQL не решает.

## Revisit when
Переезд на Postgres (OPEN, прод-подготовка) или рост каталога → заменить на pg_trgm (нечёткость
с GIN-индексом) + `SearchVector`/`SearchRank` с русским стеммингом для морфологии и масштаба.

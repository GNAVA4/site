# ADR 002: Доступ к заказам — гостевой трекинг по коду (без аккаунтов)
_Date: 2026-05-31 session 024 | Status: active_

## Context
Пользователь хотел дать покупателям возможность смотреть свои заказы/суммы и автоподстановку
данных. Встал вопрос: делать ли аутентификацию (аккаунты), и как это влияет на обязательства по
персональным данным (152-ФЗ).

## Options considered
| Option | Pros | Cons |
|--------|------|------|
| A. Код заказа (гость) | минимум ПДн, нет паролей, быстро; даёт трекинг | нет единого «списка моих заказов» |
| B. Вход по телефону (OTP) | список всех заказов + автоподстановка, без паролей | нужен канал отправки OTP, сложнее |
| C. Полные аккаунты | максимум возможностей | макс. ПДн/ответственность, пароли, безопасность входа |

## Decision
Выбран **A**: каждому заказу — случайный код (8 символов, алфавит без I/O/0/1). Проверка статуса
на /order/track/ по номеру+коду (constant-time сравнение). Автоподстановка имени/телефона — на
стороне браузера (localStorage), НЕ на сервере. — Source: [user]

## Consequences
- ✅ Почти нет новых ПДн (только токен на заказ); нет паролей/учёток → меньше обязательств и рисков
- ✅ Покупатель видит статус/состав/сумму; данные предзаполняются (приватно, в браузере)
- ✅ Апгрейд до B (вход по телефону) возможен поверх, без слома
- ⚠️ Нет «личного кабинета» со списком всех заказов (по дизайну); код надо сохранить
- ⚠️ Код = capability: кто знает номер+код, видит этот заказ (приемлемо; перебор защищён длиной кода + compare_digest)

## Confidence
HIGH — соответствует масштабу магазина и минимизирует ПДн-обязательства.

## Revisit when
Если появится много повторных клиентов, которым нужен полноценный кабинет/история — перейти к B или C.

# ADR 007 — Роль «Менеджер»: ограниченный доступ в админку

_Date: 2026-06-14 (session 039) · Status: active_

## Контекст
Владелец заходит суперпользователем (полный доступ). Нужна вторая роль — менеджер: может полностью
вести магазин (товары, категории, баннеры, заказы, настройки), но без критичных/опасных действий.

## Решение
Группа **«Менеджеры»** (создаётся data-миграцией `store/0017_manager_group.py`, применяется и на проде).
Менеджер = `is_staff=True` + членство в группе. Набор прав (24 шт.):
- **Товары / размеры / фото-галерея / баннеры** — add/change/delete/view (полный CRUD).
- **Категории** — add/change/view, **БЕЗ delete** (delete каскадит на товары → потеря каталога).
- **Заказы** — change/view, **БЕЗ add/delete** (финансовые записи + ПДн; создаются только с сайта).
- **Позиции заказа** — только view (снапшот неизменен; инлайн уже read-only).
- **Настройки сайта** — change/view (контакты/каналы/цвет/тексты = управление магазином; add/delete блок singleton'ом).
- **Пользователи и группы (auth)** — НЕ выдаём вовсе → менеджер не видит /admin/auth/ (403) и не может выдать себе суперправа.

Защита в коде (defense-in-depth, не только права): `OrderAdmin.has_delete_permission` и
`CategoryAdmin.has_delete_permission` → `return request.user.is_superuser`. Даже при ошибочной выдаче прав
удалить заказ/категорию сможет только суперпользователь.

## Провенанс
Пользователь (s039): «зайти суперпользователем + создавать обычного пользователя в админке; без критичного
типа удаления заказов; придумай что ещё ограничить». Критичным признано: удаление заказов (деньги/ПДн),
удаление категорий (каскад), управление пользователями/правами (эскалация).

## Как завести менеджера
Суперпользователь: Админка → Пользователи → Добавить → задать логин/пароль → на след. экране отметить
«Статус персонала» (is_staff) и добавить в группу «Менеджеры» (НЕ отмечать «Статус суперпользователя»).
